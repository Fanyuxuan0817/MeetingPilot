# MeetingPilot 后续实施路线图：从 Mock 到真 AI

## 第一阶段：后端“真转录”主干道 (Persistence & ML)

**目标**：让用户上传的音频真正变成数据库里的文字。

1. **数据库持久化 (PostgreSQL)**：
* 在 `backend/app/models/` 编写 SQLAlchemy 模型（Meeting, TranscriptChunk）。
* 使用 Alembic 执行迁移：`alembic revision --autogenerate`。
* **关键点**：确保 `transcript_chunks` 表包含 `start`, `end`, `speaker`, `content` 字段，并与 `meeting_id` 关联。


2. **集成核心转录服务 (Faster-Whisper)**：
* 在 `services/transcription_service.py` 中编写逻辑。
* 调用本地或 API 形式的 Whisper 模型。
* **关键点**：实现“解析 -> 存库”流程。音频上传后，立即触发转录，并将结果批量写入数据库，而不是只留在内存里。


3. **替换 Mock 接口**：
* 修改 `GET /meetings/{id}/transcripts`，使其从数据库读取数据。
* 此时，前端刷新后应能看到真实的转录内容。



---

## 第二阶段：LangGraph 多 Agent 编排 (The Brain)

**目标**：实现 PRD 中设计的智能分析能力。

1. **定义状态机 (State Definition)**：
* 在 `agents/state.py` 定义 `AgentState`（包含当前文本、已生成的摘要、待办列表等）。


2. **构建核心节点 (Nodes)**：
* **Summary Node**：将转录文本发给 LLM，输出结构化摘要。
* **Action Node**：提取任务、责任人和截止日期。
* **Conflict Node**（亮点）：分析前后矛盾的决策。


3. **编排 Graph**：
* 使用 `LangGraph` 将这些节点连起来。
* 实现“转录完成 -> 自动触发 Graph 执行”的任务流。



---

## 第三阶段：语义搜索与长期记忆 (Vector DB)

**目标**：让 AI 能够回答关于会议细节的问题。

1. **向量化 (Embedding)**：
* 在转录完成后，将每个文本块（Chunk）转换为向量。


2. **存入 Qdrant**：
* 将向量和原始文本、时间戳存入 Qdrant。


3. **实现 RAG 问答**：
* 编写 `QA Agent`。当用户提问时，先去 Qdrant 检索最相关的 3-5 个片段，再交给 LLM 生成回答。
* **前端配合**：实现“点击 AI 回答中的引用，跳转到音频对应时间点”。



---

## 第四阶段：WebSocket 实时交互 (Real-time)

**目标**：提升 AI 对话的体验。

1. **流式输出 (Streaming)**：
* 将后端 `QA Agent` 的输出改为流式返回。


2. **前端适配**：
* 在 Vue 中通过 `useWebSocket` 接收数据，实现 AI 回答逐字蹦出的效果，增强“智能感”。



---

## 第五阶段：外部集成与 UI 润色 (Final Polish)

**目标**：从“工具”变成“产品”。

1. **飞书/钉钉集成**：
* 实现 `Execution Agent`，将待办事项一键同步到飞书任务。


2. **视觉强化 (可爱风)**：
* 接入你设计的“卡通猫 Coach” UI 元素。
* 添加转录时的加载动画（Skeleton Screen）。
